# Pin Window Shadow Strategy

## 背景

当前钉住窗口已经具备真实悬浮与重钉能力，但之前把圆角和阴影直接加在图片层，导致截图内容被视觉组件化处理。实际表现包括：

- 截图边缘被圆角裁切，内容完整性受影响
- 阴影像贴在图片表面，而不是“窗口悬浮在桌面上”
- 在浅色背景下虽然能看到悬浮感，但整体更像一张带样式的卡片，而不是一张被钉住的截图

这轮目标不是继续调圆角或阴影数值，而是把视觉职责重新放回正确层级。

## 决策

### 1. 窗口层负责阴影

钉住窗口的悬浮感由原生窗口阴影承担：

- 在创建 pin window 时开启 `hasShadow`
- 保持 `transparent: true`、`frame: false`、`alwaysOnTop: true`
- 不再依赖图片层或容器层的 CSS `box-shadow`

这样阴影来自系统窗口层，而不是画在截图表面。

### 2. 容器层不承担装饰

容器层只保留最小结构职责：

- 不设置圆角
- 不设置描边
- 不设置自定义阴影
- 不额外制造“卡片感”

这轮明确放弃容器层的装饰性样式，避免出现视觉冗余。

### 3. 图片层保持原始截图

图片层只负责展示截图内容：

- 不加圆角
- 不加阴影
- 不加描边
- 保持截图边缘、文字和 UI 元素的原始视觉

这样截图内容不会因为视觉包装被误改。

## 实现范围

这轮只允许修改：

- `public/preload/pinWindowManager.cjs`
- `public/pin-window.html`
- 最相关测试

不改：

- 自定义截图链路
- 记录页结构
- 重钉状态持久化逻辑
- 其他页面样式

## 验证重点

1. 点击“重新钉住”后，窗口仍能正常唤回。
2. 钉住图片边缘不再被人为圆角裁切。
3. 桌面上能够通过窗口原生阴影看出钉住位置。
4. 构建产物仍可正常加载 pin window。

## 风险与接受项

- 原生窗口阴影强度由系统决定，跨平台观感不会完全一致。
- 这轮不追求精细可控的设计化阴影，只追求“足够看出位置”的系统级悬浮感。
- 如果后续系统原生阴影仍不够明显，再单独讨论是否引入轻量容器层辅助样式，但不在本轮实现范围内。
