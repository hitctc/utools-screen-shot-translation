# 记录页列数与卡片密度优化设计

日期：2026-04-11
分支：`feature/custom-capture-pin`

## 1. 目标

当前 `钉住记录` 页虽然已经使用瀑布流，但卡片视觉体积偏大，导致记录数量稍少时也会显得页面松散，不符合之前确认的紧凑缩略图浏览体验。

本轮目标：

- 记录页默认以 `3` 列展示
- 用户可在记录页顶部直接调整列数
- 调整方式使用进度条 / 滑块，范围 `3 ~ 6`
- 用户选择的列数需要持久化，下次进入仍保持
- 在列数增加后，卡片也要同步收紧，不再保留当前偏重的大卡片观感
- 移动或窄窗口场景仍保留自动降列，避免把缩略图压到不可用

## 2. 范围

本轮只优化记录页的布局密度和列数调节交互，不修改以下内容：

- 记录页的删除、重钉、空态、warning 语义
- 设置页结构
- 主流程 `截图 -> 翻译 -> 钉住`
- 保存目录与总清单数据结构

## 3. 交互设计

### 3.1 顶部工具条

在 `钉住记录` 页顶部增加一个工具条区域，和现有标题区保持同一视觉体系。

工具条包含：

- 当前记录总数
- 列数调节滑块
- 当前列数字样
- 设置按钮

列数调节语义：

- 默认值：`3`
- 最小值：`3`
- 最大值：`6`
- 步长：`1`
- 拖动后立即生效
- 生效后立即持久化到 `UiSettings`

展示建议：

- 标题文案：`列数`
- 右侧实时显示：`3 列`、`4 列` 这类状态文案
- 不额外做“确认”按钮，减少一次交互

### 3.2 响应式规则

桌面宽度下优先使用用户设定的列数。

窄屏规则：

- 宽度足够时按用户设置 `3 ~ 6` 列展示
- 中等宽度自动下压到 `2` 列
- 很窄时下压到 `1` 列

也就是说，用户设置表示“桌面首选列数”，而不是强制在任何窗口宽度下都死撑该列数。

## 4. 视觉与布局调整

### 4.1 列布局

继续保留瀑布流思路，不切换回规则栅格。

实现上优先继续使用 CSS multi-column：

- 通过 CSS 变量控制 `column-count`
- `records-grid` 根据当前列数设置变量
- 卡片继续使用 `break-inside: avoid`

这样改动最小，不需要重写整套列表布局。

### 4.2 卡片密度

当前卡片偏大的主要原因：

- 卡片外壳 padding 偏厚
- 图片被强制为固定 `4 / 5` 比例
- 预览、元信息、操作按钮的纵向空隙偏大

本轮收紧原则：

- 缩小卡片外边距和内边距
- 图片按真实比例展示，不再强行固定 `aspect-ratio`
- 操作区保持存在，但按钮尺寸略收紧
- 叠层信息保留编号和“重钉”提示，不额外增加视觉负担

目标观感：

- 一屏能看到更多记录
- 图片高宽比自然变化，瀑布流更明显
- 仍然保持可点击、可删除、可识别日期与编号

## 5. 数据与状态

### 5.1 UI 设置扩展

`UiSettings` 新增字段：

- `recordsColumnCount`

默认值：

- `3`

有效范围：

- `3 ~ 6`

归一化规则：

- 非法值回退到 `3`
- 浮点值取整
- 超出范围时裁切到区间内

### 5.2 持久化

列数属于纯 UI 偏好，继续保存在 `dbStorage` 对应的 UI 设置里，不放进插件业务设置。

原因：

- 它只影响界面展示
- 和主题模式、窗口高度同类
- 不应混入翻译或保存逻辑设置

## 7. 验证

最小验证项：

1. 默认进入 `钉住记录` 时显示 `3` 列
2. 拖动滑块到 `4 / 5 / 6` 时立即改变布局
3. 重新进入插件后仍保持上次所选列数
4. 窄窗口下即使用户选择 `6` 列，也会自动降列，不出现不可读缩略图
5. 卡片视觉收紧后，删除和重钉交互不受影响

## 8. 非目标

本轮不做：

- 列数设置放到设置页
- 自定义卡片尺寸
- 拖拽排序
- 筛选、搜索、分组
- 预览大图
