# uTools Screen Shot Translation 迁移设计

## 1. 目标

本轮目标不是实现完整的“截屏 -> 翻译 -> 钉住”能力，而是把当前仓库整理成一个语义正确、结构完整、可以继续开发的 uTools 插件工程。

这次工作要完成三件事：

- 以 `utools-my-quick-bookmarks` 为参考，完整迁移它已经跑通的工程结构。
- 清理旧书签业务，避免继续保留 `bookmarks` 相关页面、模块、状态和持久化语义。
- 把当前仓库替换成“截屏 / 翻译 / 钉住”的空骨架插件，并保留设置页，方便后续接入真实能力。

## 2. 本轮范围

### 2.1 要做的事

- 复制参考项目的完整工程结构到当前仓库。
- 把插件身份替换为新的截屏翻译插件。
- 把首页替换为单入口三步流骨架。
- 保留设置页，但改成新插件相关的配置骨架。
- 保留适合当前插件的 `preload` 桥接结构。
- 删除旧书签专用模块、文案、测试和持久化 key。
- 跑最相关测试和构建，确认迁移后的骨架工程可用。

### 2.2 这轮不做的事

- 不接入真实截屏能力。
- 不接入真实 OCR 或翻译服务。
- 不实现真正的图片钉住到屏幕。
- 不引入多 feature、多窗口或多入口插件结构。
- 不做与本轮迁移无关的重构。

## 3. 插件身份

本轮迁移后，插件基础身份固定为：

- 显示名：`uTools Screen Shot Translation`
- `feature.code`：`screen-shot-translation`
- `cmds`：
  - `截屏&翻译`
  - `截屏&翻译&钉住`

`public/plugin.json`、前端入口、README、测试命名和注释里的旧书签语义都必须同步替换，不能只改其中一部分。

## 4. 总体方案

本轮采用“最小迁移版”方案：

- 保留参考项目已经跑通的 `uTools + Vite + Vue` 工程组织方式。
- 保留单 `feature`、单 `App.vue` 状态机、单 `preload/services.js` 桥接的主结构。
- 不保留旧书签业务目录和逻辑。
- 用新的业务目录和状态模型替换旧书签目录。

选择这个方案的原因：

- 它最符合这轮“完整迁移并仔细全局替换”的目标。
- 替换范围清晰，最不容易漏改插件身份。
- 现在先把工程语义切干净，比提前做结构重构更重要。

## 5. 页面与交互设计

## 5.1 首页结构

首页改成单入口三步流，不做复杂向导，而是一个清晰的流程骨架页。

首页会展示 3 个步骤卡片：

1. `截屏`
2. `翻译`
3. `钉住`

每个步骤卡片都要有：

- 标题
- 当前状态文案
- 简短说明
- 骨架级操作按钮或提示

首页还需要保留：

- 进入设置页按钮
- 错误提示区域
- 初始加载状态
- 当前步骤状态或流程状态提示

### 5.2 首页状态模型

这轮首页只保留骨架需要的最小状态，不提前引入真实业务复杂度。

建议状态包括：

- `currentView`
  - `home | settings`
- `bootstrapped`
  - 是否已经完成插件基础初始化
- `processing`
  - 是否处于某个骨架动作执行中
- `currentStep`
  - `capture | translate | pin`
- `captureState`
  - 当前截屏占位状态
- `translationState`
  - 当前翻译占位状态
- `pinState`
  - 当前钉住占位状态
- `homeError`
  - 首页级错误提示
- `uiSettings`
  - 主题模式、窗口高度和新插件的基础配置

这些状态只服务于三步流骨架显示，不承诺真实截图、翻译或钉住已经可用。

### 5.3 首页交互

首页交互先做成占位链路：

- 用户点击“开始截屏”后，只推进到下一阶段的骨架状态。
- 用户点击“开始翻译”后，只更新翻译结果占位状态。
- 用户点击“开始钉住”后，只更新钉住状态占位信息。

页面要明确提示这些是骨架动作，避免形成“功能已接入”的误解。

## 5.4 设置页结构

设置页保留，但彻底去掉书签相关内容。

设置页本轮保留以下内容：

- 翻译相关基础配置骨架
  - 例如语言方向、翻译服务说明占位
- 显示相关配置骨架
  - 例如钉住结果的默认表现说明
- 主题模式
- 插件窗口高度
- 返回首页按钮

设置页本轮不保留：

- 书签路径
- 路径选择
- 书签刷新
- 最近打开和打开次数展示开关
- 任何 Chrome 或书签说明文案

设置页的配置项要以“后续能自然接到真实功能”为原则，避免放入大量纯伪开关。

## 6. 目录与模块设计

参考项目中的 `src/bookmarks/` 会整体替换为新业务目录，建议命名为：

- `src/screenTranslation/`

这个目录下至少会有：

- `HomeView.vue`
- `SettingsView.vue`
- `types.ts`

根据骨架实现需要，可以再补少量与步骤卡片或状态展示相关的组件，但不保留旧书签卡片组件。

会删除的旧模块包括：

- 搜索模块
- 排序模块
- 键盘导航模块
- 书签卡片与区块组件
- 旧书签主题状态格式化里的书签专用文案

保留但会改造的模块包括：

- `src/App.vue`
- `src/main.css`
- `src/main.js`
- `src/bootShell.js`

`src/Hello`、`src/Read`、`src/Write` 这类模板目录如果参考项目里还存在，本轮会一并清理，避免继续携带无关历史模板。

## 7. preload 与本地桥接边界

本轮仍然保留 `public/preload/services.js` 作为渲染层访问本地能力的统一入口。

### 7.1 本轮保留的桥接能力

- 读取插件 UI 设置
- 保存插件 UI 设置
- 读取截屏翻译插件的基础配置
- 保存截屏翻译插件的基础配置
- 后续真实能力预留的接口壳

### 7.2 本轮删除的桥接能力

- Chrome 书签默认路径计算
- 书签文件读取
- 书签缓存
- 置顶状态
- 最近打开和打开次数
- 打开书签 URL

### 7.3 状态持久化

所有 `quick-bookmarks-*` 相关 key 都要替换掉。

新的 key 需要改成与当前插件匹配的语义，例如：

- `screen-shot-translation-settings`
- `screen-shot-translation-ui-settings`

最终命名以实现时统一风格为准，但必须满足两个条件：

- 不再保留 `bookmarks`、`chrome`、`quick-bookmarks` 等旧语义
- 配置项与 UI 项职责分离，避免一个 key 承担所有状态

## 8. 需要删除或替换的旧能力

下面这些旧能力要被视为必须清理项：

- 书签读取与解析
- 书签搜索
- 拼音搜索
- 键盘上下选择
- 回车打开书签
- 置顶逻辑
- 最近打开与打开次数
- 书签缓存秒开
- 书签路径恢复默认值
- 任何 Chrome Bookmarks 文件说明

如果迁移后还有这些概念残留在代码、注释、README、测试或 UI 文案中，就说明替换不完整。

## 9. 文件级改动范围

本轮预计会改动或新增这些层级：

- 工程身份层
  - `README.md`
  - `package.json`
  - `public/plugin.json`
- 前端层
  - `src/App.vue`
  - `src/main.css`
  - `src/main.js`
  - `src/bootShell.js`
  - `src/screenTranslation/*`
- preload 层
  - `public/preload/services.js`
  - `public/preload/localState.cjs`
  - 删除旧 `chromeBookmarks.cjs`
- 测试层
  - `tests/preload/*`

如果参考项目里还有与当前插件无关的模板文件、历史测试或无用资源，本轮一并清理。

## 10. 验证策略

按“小到大”的顺序验证：

1. 先做全局文本检查，确认旧书签语义已基本清除。
2. 跑最相关测试，确认新的本地设置与纯函数逻辑可用。
3. 跑 `npm run build`，确认 uTools 插件骨架能构建。

这轮最小验证目标是：

- 工程可以安装依赖
- 相关测试可跑
- 构建成功
- 打开插件后能看到新的三步流骨架和设置页

## 11. 风险与边界

### 11.1 已知风险

- 本轮只做骨架，不会提供真实截屏翻译能力。
- 后续接入真实截屏、OCR、翻译服务和钉住窗口时，`preload` 和主状态机还会继续扩展。
- 如果迁移时全局替换不彻底，容易留下旧插件名称、旧测试或旧持久化 key。

### 11.2 控制策略

- 先完整迁移参考结构，再集中做全局替换。
- 删除旧业务模块，而不是留着不用。
- 用搜索检查旧关键词残留，例如 `bookmark`、`bookmarks`、`chrome`、`quick-bookmarks`。
- 只保留和当前插件骨架直接相关的测试与注释。

## 12. 实施后的交付状态

本轮完成后，当前仓库应满足：

- 它已经是一个结构完整的 uTools 插件工程，而不是一个只有 README 的目录。
- 打开插件时展示的是“截屏 / 翻译 / 钉住”流程骨架，而不是书签页面。
- 设置页语义已经切换到新插件方向。
- 旧书签逻辑、旧书签测试和旧书签文案被清理掉。
- 后续可以在这个骨架上继续实现真实截屏、翻译和钉住能力。
