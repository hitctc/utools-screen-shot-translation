# uTools Screen Shot Translation 设计稿

## 1. 背景与目标

这轮要把当前“截屏 -> 翻译 -> 钉住”的骨架工程收口成一个可继续实现的产品方案，而不是继续停留在占位页面。

目标只有 3 个：

- 明确插件的 3 个入口及各自职责。
- 明确设置页、记录页、失败结果页和无页面主流程的边界。
- 明确保存目录、总清单 JSON、重钉位置和删除规则的数据模型，确保后续实现时不再反复改协议。

## 2. 本轮范围

### 2.1 需要覆盖的设计范围

- `plugin.json` 的 3 个入口设计
- 无页面主流程的成功与失败行为
- `钉住记录` 主页结构与交互
- `设置` 页字段与约束
- 结果图片保存策略
- 记录总清单 JSON 结构
- 重新钉住与删除流程
- 失败结果页的信息层级

### 2.2 这轮不展开的实现细节

- 不在设计稿里绑定具体 UI 视觉风格
- 不展开百度接口签名的代码实现
- 不展开钉住窗口的底层 Electron 参数细节
- 不引入搜索、筛选、分页、拖拽调整位置等增强功能
- 不在记录页中再放“开始截屏”按钮

## 3. 产品结构总览

插件最终采用“1 个无页面主流程 + 2 个页面入口 + 1 个失败结果页”的结构。

### 3.1 入口一：`截屏翻译钉住`

这是主功能入口，但不是页面入口。

触发后立即执行：

1. 调用 uTools 截屏能力
2. 把截图交给百度图片翻译
3. 得到翻译结果图后，按原截图位置钉住
4. 根据设置决定是否保存结果图

成功时：

- 不打开任何页面
- 直接完成钉住

失败时：

- 打开失败结果页
- 向用户说明当前失败在哪一步、下一步该做什么

### 3.2 入口二：`钉住记录`

这是主页入口。

主页职责只保留两类：

- 展示已经保存到用户目录中的结果图片
- 提供重新钉住、删除、进入设置的入口

主页不承担主流程启动职责，不在这里再放“开始截屏”按钮。

### 3.3 入口三：`设置`

这是设置页直达入口。

用户可以：

- 直接通过 `设置` 指令打开设置页
- 也可以从 `钉住记录` 主页进入设置页
- 失败结果页也允许跳转到设置页

## 4. 页面与状态设计

## 4.1 `钉住记录` 主页

主页展示形式确定为“瀑布流缩略图网格”。

原因：

- 截图天然是非固定比例
- 固定比例卡片会明显裁切或留白
- 记录页的首要目标是快速浏览历史结果图

### 4.1.1 数据来源

主页只读取“用户设置的保存目录”中的结果图片，不额外维护第二套独立记录库。

为了让主页可以直接渲染排序、编号和位置数据，保存目录根目录下维护一份总清单 JSON 文件。

### 4.1.2 排序规则

- 按结果图片生成并保存的时间倒序排列
- 最新的记录排在最前面

### 4.1.3 卡片信息

每张缩略图卡片至少展示：

- 缩略图
- 编号
- 结果生成日期
- 删除按钮

编号的目标不是稳定主键，而是帮助用户快速感知：

- 当前大约有多少张图
- 自己浏览到列表的大概位置

### 4.1.4 卡片点击行为

点击缩略图默认行为不是预览，而是“重新钉住到屏幕”。

重新钉住时要回到这张图片“上一次被钉住时的位置”，因此记录里必须保留最近一次钉住的位置信息。

### 4.1.5 删除行为

卡片上提供删除按钮。

删除时行为受设置项控制：

- 如果“删除图片前需要二次确认”开启，则先确认再删
- 如果关闭，则直接删除

无论哪种模式，真正删除时都必须同时处理：

- 图片文件
- 总清单 JSON 中对应的记录

## 4.2 设置页

设置页只保留真实能力会用到的字段，不保留骨架时代的伪配置。

### 4.2.1 翻译方向

翻译方向采用 3 选 1：

- `自动`
- `英 -> 中`
- `中 -> 英`

原因：

- 默认自动检测最省心
- 仍然允许用户在需要时强制指定方向

### 4.2.2 保存结果图片

保存由两个设置共同决定：

- `是否保存翻译结果图片`
- `保存目录`

只有在以下条件同时满足时，主流程才会保存结果图：

- 保存开关开启
- 保存目录已设置

如果保存开关开启但目录为空：

- 设置页要显示显著警告
- 主流程执行时不继续成功链路，而是打开失败结果页并提示用户先补全设置

### 4.2.3 删除前二次确认

设置页增加一个布尔开关：

- `删除图片前需要二次确认`

默认建议值：`开启`

原因：

- 记录页是缩略图瀑布流，误触风险存在
- 删除是不可逆动作

### 4.2.4 可见辅助信息

设置页可展示但不必复杂化的辅助信息包括：

- 当前保存目录路径
- 当前目录下已保存结果图数量
- 当前配置是否完整可用

## 4.3 失败结果页

失败结果页只在主流程失败时打开，成功时不出现。

### 4.3.1 页面职责

- 告诉用户失败发生在哪个阶段
- 用用户可理解的话解释失败原因
- 提供最短恢复路径

### 4.3.2 失败阶段分类

失败原因按阶段归类即可，不直接暴露底层异常栈：

- 截屏取消
- 翻译失败
- 保存配置不完整
- 保存失败
- 钉住失败
- 重新钉住失败

### 4.3.3 建议动作

不同失败类型对应的默认动作：

- 截屏取消：关闭
- 翻译失败：重试、关闭
- 保存配置不完整：前往设置、关闭
- 保存失败：前往设置、关闭
- 钉住失败：重试、关闭
- 重新钉住失败：重试、关闭

## 5. 数据模型设计

## 5.1 设置存储模型

设置仍然走 `preload -> window.services -> utools.dbStorage` 这一层统一桥接。

建议把插件设置扩展为至少包含：

- `translationMode`
  - `auto | en-to-zh | zh-to-en`
- `saveTranslatedImage`
  - `boolean`
- `saveDirectory`
  - `string`
- `confirmBeforeDelete`
  - `boolean`

UI 设置与插件设置仍建议分离，主题和窗口高度继续留在 UI 设置中。

## 5.2 保存目录结构

用户选择的保存目录根目录中包含：

- 结果图片文件
- 一份总清单 JSON

总清单文件放在保存目录根目录，不放隐藏子目录。

建议命名：

- `.screen-translation-records.json`

如果实现阶段需要兼容性更强的命名，也可以在不改变“根目录单文件清单”前提下微调名称。

## 5.3 总清单 JSON 结构

建议结构如下：

```json
{
  "version": 1,
  "updatedAt": "2026-04-10T22:30:00.000Z",
  "records": [
    {
      "id": "rec_20260410_223000_ab12cd",
      "imageFilename": "20260410-223000-en-to-zh.png",
      "createdAt": "2026-04-10T22:30:00.000Z",
      "lastPinnedAt": "2026-04-10T22:31:15.000Z",
      "lastPinBounds": {
        "x": 120,
        "y": 84,
        "width": 860,
        "height": 420
      }
    }
  ]
}
```

### 5.3.1 字段说明

- `version`
  - 清单版本号，便于后续兼容升级
- `updatedAt`
  - 清单最后一次写入时间
- `id`
  - 记录主键
- `imageFilename`
  - 结果图片文件名，和目录路径拼接后可定位图片
- `createdAt`
  - 结果图片生成并保存的时间
- `lastPinnedAt`
  - 最后一次钉住成功时间
- `lastPinBounds`
  - 最近一次成功钉住的位置与尺寸

## 5.4 脏数据处理

主页读取总清单时需要允许自修复。

典型情况：

- 清单有记录，但图片文件已不存在
- 图片文件还在，但清单字段缺失或损坏

建议策略：

- 渲染时跳过无效记录
- 在下一次成功保存、删除或主动整理时回写清单，清掉脏记录

本轮不额外设计“目录扫描补录孤儿图片”，避免复杂度继续上升。

## 6. 主流程设计

## 6.1 `截屏翻译钉住` 成功路径

推荐的流程边界如下：

1. 进入指令
2. 隐藏主窗口，避免影响截图体验
3. 调起 uTools 截屏能力
4. 获取截图原始数据以及原始位置/尺寸信息
5. 调百度图片翻译 API 获取实景回填结果图
6. 在原截图位置创建钉住结果
7. 如果保存开关开启且目录已设置，则保存结果图并更新总清单
8. 整体成功后退出，不打开页面

这里有一个实现层推断：

- “按原位置钉住”大概率需要基于 `utools.createBrowserWindow(...)` 创建独立窗口，再通过 Electron `BrowserWindow` 能力做无边框、置顶、固定位置展示

这不是当前设计要锁死的唯一实现方式，但它是目前基于官方窗口文档最合理的候选路径。

## 6.2 `截屏翻译钉住` 失败路径

一旦以下任一步骤失败，就打开失败结果页：

- 用户取消截屏
- 百度翻译失败
- 保存开关已开启但目录未设置
- 保存目录不可写
- 图片保存失败
- 总清单写入失败
- 钉住窗口创建失败

## 6.3 `钉住记录` 重新钉住路径

点击记录卡片时：

1. 根据 `imageFilename` 找到结果图片
2. 读取 `lastPinBounds`
3. 在对应位置重建钉住窗口
4. 成功后更新 `lastPinnedAt`
5. 如果位置或尺寸发生必要修正，也同步回写 `lastPinBounds`

如果图片文件已不存在或位置数据不可用：

- 不静默失败
- 打开失败结果页或给出明确失败提示

## 7. 技术边界与模块落点

## 7.1 preload 边界

所有涉及本地文件、系统能力、网络请求和 uTools API 的真实能力，都应继续优先放进 `public/preload/`。

原因：

- 当前项目已经采用 `window.services` 暴露桥接能力
- 官方文档明确 `preload` 可以调用 Node.js 原生能力与 Electron 渲染进程 API，并通过 `window` 暴露给前端

因此后续建议新增的桥接能力包括：

- 读取与保存插件设置
- 选择保存目录
- 读取总清单
- 写入总清单
- 保存图片文件
- 删除图片文件
- 执行主流程
- 执行重新钉住

## 7.2 前端边界

前端页面层只负责：

- 渲染记录页
- 渲染设置页
- 渲染失败结果页
- 调用 `window.services` 暴露的业务接口

前端不直接散落：

- 文件系统读写
- 百度请求签名
- 钉住窗口创建细节
- 目录扫描逻辑

## 8. 最小实现范围

第一轮实现只做到以下闭环：

- 3 个入口能正确进入对应流程或页面
- 主流程成功时直接钉住，失败时打开结果页
- 设置页支持真实可用字段
- 记录页支持瀑布流展示、编号、日期、删除、点击重钉
- 保存目录根目录维护一份总清单 JSON

明确不纳入第一轮：

- 图片预览大图
- 搜索与筛选
- 分页
- 拖拽调整钉住位置后自动持久化
- 批量删除
- 目录迁移向导

## 9. 验证建议

后续实现阶段建议按下面顺序验证：

1. 设置存储与总清单纯函数
2. 文件读写与清单更新
3. 主流程服务层
4. 记录页和设置页界面
5. `npm test`
6. `npm run build`
7. uTools 内手动 smoke test

手动 smoke 的核心路径：

- 通过 `设置` 打开设置页，验证保存开关、目录、翻译方向、删除确认开关
- 通过 `钉住记录` 进入主页，验证空状态与瀑布流
- 通过 `截屏翻译钉住` 执行主流程，验证成功无页面、失败进结果页
- 从记录页点击缩略图重新钉住
- 验证删除在开关打开/关闭两种模式下的行为

## 10. 文档依据与推断说明

本设计稿依赖以下官方资料：

- uTools 基础文档：插件基础配置、`preload`、屏幕/窗口/数据存储能力总览
- uTools `preload` 文档：确认 `preload` 可调用 Node.js 与 Electron API，并通过 `window` 暴露给前端
- uTools `window` 文档：确认可调主窗口高度、创建独立窗口、打开目录选择框
- uTools `dbStorage` 文档：确认设置项可继续走键值对本地存储
- 百度翻译开放平台图片翻译文档与产品页：确认图片翻译支持语种自动检测与译文实景回填

需要明确标注为“实现层推断”的点：

- “钉住结果图”很可能需要通过 `utools.createBrowserWindow(...)` + Electron `BrowserWindow` 能力实现
- “截屏得到原始位置与尺寸”需要在真正实现时再对照 uTools 屏幕能力文档验证返回结构，当前设计只锁定需要这类数据，不锁死具体字段名

这两个点在开始编码前必须再次核对官方文档或实际运行结果，不能凭设计稿直接臆造接口细节。
